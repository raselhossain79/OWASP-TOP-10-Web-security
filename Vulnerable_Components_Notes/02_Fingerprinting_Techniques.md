# 02 — Fingerprinting Techniques

Goal of this file: identify exactly what software (and ideally what version)
a target is running, using both manual inspection and tool-assisted
fingerprinting. Every command below is broken down flag by flag — never run
a command you can't explain.

## 1. Manual fingerprinting first (do this before any tool)

Tools can be wrong, rate-limited, or blocked. Manual checks are free, quiet,
and often give you the first lead.

| Source | What to look for | Why it works |
|---|---|---|
| HTTP response headers | `Server`, `X-Powered-By`, `X-AspNet-Version`, `Via` | Many servers/frameworks self-report their identity and version by default |
| Error pages / stack traces | Framework name + version in a 500 error | Misconfigured apps leak this in debug mode (this is literally how the PortSwigger lab in File 05 is solved) |
| HTML source comments | `<!-- Generated by WordPress 6.2 -->`-style comments, generator meta tags | CMS platforms and static site generators often self-identify |
| JS/CSS file paths | `/wp-content/`, `/sites/all/modules/`, `/_next/static/`, versioned filenames like `jquery-1.7.2.min.js` | Path conventions and filenames are strong fingerprints for frameworks/CMSs and their bundled libraries |
| `robots.txt` / `sitemap.xml` | CMS-specific disallow paths (`/wp-admin/`, `/typo3/`) | Default install paths vary by platform |
| Favicon hash | Match against known favicon hash databases (e.g. via Shodan's `http.favicon.hash`) | Many off-the-shelf products ship a distinctive default favicon |
| Cookies | Cookie names like `JSESSIONID`, `PHPSESSID`, `laravel_session` | Session cookie naming conventions are framework-specific defaults |

Always cross-check at least two independent signals before trusting a version
number — headers can be deliberately spoofed or stripped by a reverse proxy/WAF.

## 2. WhatWeb

WhatWeb is a Ruby-based fingerprinting tool with a large signature database
covering CMSs, web servers, JS libraries, analytics platforms, and more.

```bash
whatweb -a 3 -v --color=never -t 5 https://target.example.com
```

Flag-by-flag breakdown:

- `whatweb` — invokes the tool.
- `-a 3` — sets the **aggression level** to 3 ("aggressive"). WhatWeb has
  4 aggression levels (0–3); level 1 is passive (single request, default),
  level 3 makes additional requests per plugin to confirm tentative matches
  (e.g. requesting a known CMS-specific file path to confirm a guess made from
  the homepage alone). Higher aggression = more accurate but more noisy/requests
  sent — relevant if stealth matters on the engagement.
- `-v` — **verbose** output. Shows the actual evidence string (e.g. the exact
  header or HTML snippet) WhatWeb matched on, not just the plugin name. This is
  essential for screenshotting/quoting evidence in a report.
- `--color=never` — disables ANSI color codes in output, which keeps logs/output
  files clean when piping to a file or report.
- `-t 5` — sets the number of concurrent **threads** to 5. Controls scan speed
  vs. load on the target; lower this on fragile or rate-limited targets.
- `https://target.example.com` — the target URL.

Other flags worth knowing:

- `-i targets.txt` — read a list of targets from a file (useful for testing
  many subdomains/hosts in one run).
- `--log-json=output.json` — writes structured JSON output, useful for feeding
  into other tooling or for appendices in a report.
- `-p proxy_ip:port` — routes requests through a proxy (e.g. Burp) so you can
  see/log every request WhatWeb sends, which matters for transparency in
  client-facing engagements and for catching any unexpected behavior.

**Real-world note:** WhatWeb's signature database can lag behind the newest
software releases. A "no match" or generic match does not mean the target has
no fingerprintable software — it may mean the signature database is outdated.
Always fall back to manual checks (Section 1) when WhatWeb comes back empty.

## 3. Wappalyzer

Wappalyzer identifies the technology stack of a website (CMS, JS frameworks,
analytics, web servers, CDNs, e-commerce platforms) primarily via passive
analysis of HTML, headers, cookies, and JS globals. It's available as a
browser extension, a CLI tool, and an API.

CLI usage breakdown:

```bash
wappalyzer https://target.example.com --pretty
```

- `wappalyzer` — invokes the CLI tool (Node.js-based; installed via `npm install -g wappalyzer` or run via the `wappalyzer-cli` npm package, depending on which fork/build you have).
- `https://target.example.com` — the target URL.
- `--pretty` — formats the JSON output with indentation for human readability
  instead of a single minified line.

Other flags worth knowing:

- `--user-agent="<string>"` — sets a custom User-Agent. Some sites serve
  different markup/headers to bot-like User-Agents vs. browser User-Agents,
  so testing with both a browser UA and the tool's default UA can surface
  different results.
- `--delay=<ms>` — adds a delay between requests when crawling multiple pages,
  reducing load/noise on the target.
- `--no-redirect` — disables following HTTP redirects, useful when you want
  to fingerprint a redirect/landing page itself (e.g. a WAF block page) rather
  than the final destination.
- `--recursive --max-depth=<n>` — crawls linked pages up to a given depth
  instead of just the single URL, which surfaces technologies only loaded on
  specific pages (e.g. a checkout page using a different JS payment library
  than the homepage).

**Browser extension equivalent:** in practice, most testers use the
Wappalyzer browser extension during manual recon rather than the CLI, simply
clicking the icon to see the detected stack live while browsing. The CLI
matters when you need scriptable, repeatable output across many hosts (e.g.
external attack surface assessments covering dozens of subdomains).

**Real-world note:** Wappalyzer's detections are confidence-scored internally,
but the public output often just shows a name with no version, or a version
range. Treat a Wappalyzer hit as a *lead* to confirm manually (Section 1) or
via Nmap version detection (Section 4) before treating it as a confirmed
version for CVE research.

## 4. Nmap version detection

Nmap's `-sV` engine and the NSE `http-*` scripts go beyond port-state
scanning to actively probe and identify service versions — including, with
the right scripts, web application/server versions.

```bash
nmap -sV --version-intensity 7 -p 80,443 --script=http-headers,http-server-header,http-title target.example.com
```

Flag-by-flag breakdown:

- `nmap` — invokes the tool.
- `-sV` — enables **service/version detection**. After the normal port scan
  identifies open ports, Nmap sends a series of probes from its
  `nmap-service-probes` database and matches the responses against known
  service signatures to determine the software name and version running on
  each open port (not just "port 443 is open" but "this is nginx 1.18.0").
- `--version-intensity 7` — sets how many probes Nmap is willing to try per
  port, on a scale of 0–9 (default is 7). Lower values (e.g. 0–2) only try
  the most common/likely probes for speed and stealth; higher values (up to 9)
  try every probe in the database for maximum accuracy at the cost of more
  traffic. `7` is a reasonable middle ground for an authorized web app engagement
  where you want solid version data without maxing out probe noise.
- `-p 80,443` — restricts the scan to ports 80 and 443 (HTTP/HTTPS), since this
  is a web application engagement and you don't need to re-scan the full port
  range if it's already been done in an earlier recon phase.
- `--script=http-headers,http-server-header,http-title` — runs specific
  **Nmap Scripting Engine (NSE)** scripts:
  - `http-headers` — retrieves and displays the full set of HTTP response
    headers, which often contain `Server`, `X-Powered-By`, and similar
    version-revealing headers.
  - `http-server-header` — specifically extracts and isolates the `Server`
    header value, useful when you only care about that one header across
    many hosts.
  - `http-title` — grabs the HTML `<title>` tag, which can sometimes reveal
    CMS/admin panel identity (e.g. default install pages often say things
    like "Welcome to nginx!" or a specific CMS's default title).
- `target.example.com` — the target host.

Other flags/scripts worth knowing:

- `-O` — OS detection. Pairing OS fingerprinting with service version data
  helps you reason about whether an OS-level package manager would have
  patched a given service (e.g. Debian/Ubuntu backport vs. upstream version
  numbering can differ, which matters for accurate CVE matching — see File 04).
- `--script=http-enum` — checks for common files/directories (admin panels,
  backup files, install scripts) that can confirm a CMS/platform identity.
- `--script-args http.useragent="<string>"` — sets a custom User-Agent for the
  HTTP NSE scripts, similar in purpose to the Wappalyzer `--user-agent` flag.
- `-Pn` — skips host discovery (ping) before scanning. Necessary when the
  target blocks ICMP/ping but is still reachable on the web ports — common
  behind cloud load balancers and WAFs.
- `-oA <basename>` — outputs results in all three formats (normal, XML,
  grepable) with the given base filename, useful for keeping a clean audit
  trail of raw scan evidence for the report appendix.

**Real-world note:** Nmap's `-sV` is most reliable for the underlying web
server (Apache, nginx, IIS) and other network-level services. It is generally
*not* reliable for identifying application-framework or JS library versions
(that's WhatWeb/Wappalyzer's job) — use Nmap for the infrastructure layer and
the other two tools for the application layer. Combining all three gives you
overlapping, cross-validated evidence rather than relying on a single source.

## 5. Cross-validating before moving to CVE research

Before you take any version string into File 04's CVE workflow:

1. Confirm it with at least two independent methods from this file (e.g.
   WhatWeb signature + manual header check, or Nmap NSE + HTML source comment).
2. Check the date/last-modified evidence on the page/asset if available —
   a version string from a long-cached CDN asset might not reflect what's
   actually deployed right now.
3. Note explicitly in your working notes which method produced which version
   string, so your report evidence trail is defensible if the client disputes
   a finding.
