# 07 — Cloud Storage and Infrastructure Misconfiguration

## What it is

Cloud infrastructure misconfiguration covers insecure settings in cloud-hosted resources —
primarily object storage (AWS S3, Google Cloud Storage, Azure Blob Storage), but also IAM
over-permissioning, exposed management consoles/metadata endpoints, and insecure default
networking rules (overly open security groups). This is one of the fastest-growing
misconfiguration categories because organizations now manage infrastructure as code at scale,
and a single bad template or a single manually-toggled "make this public for testing" setting
can expose enormous amounts of data instantly.

## Part A: Public/Misconfigured Cloud Storage Buckets

### Common misconfiguration patterns

1. **Public read access (List + Get)** — the bucket's access control policy (ACL) or bucket
   policy grants the `AllUsers` / `AllAuthenticatedUsers` group (AWS) or "allUsers" (GCS) read
   and/or list permissions, meaning anyone on the internet can enumerate and download every
   object.
2. **Public write access** — even more severe: anyone can upload/overwrite objects, which can be
   used for defacement, malware hosting, or supply-chain attacks if the bucket serves
   production assets (e.g., a JS file referenced by the main website).
3. **Predictable bucket naming** — buckets named after the company, product, or environment
   (`companyname-backups`, `companyname-prod-db-dumps`) are trivially guessable, meaning
   misconfigured permissions on them are discoverable without any prior reconnaissance.
4. **Bucket policy vs. ACL conflicts** — a restrictive bucket policy can be silently overridden
   by a permissive object-level ACL (or vice versa), creating an inconsistent security posture
   that's easy to miss with a quick console glance.

### Manual testing, piece by piece (AWS S3 example)

Step 1 — Test anonymous list access:

```bash
aws s3 ls s3://target-bucket-name --no-sign-request
```

- `aws s3 ls` — the AWS CLI's S3 "list" subcommand.
- `s3://target-bucket-name` — the target bucket URI.
- `--no-sign-request` — explicitly tells the AWS CLI to send the request **without any AWS
  credentials or signature**, simulating a fully anonymous, unauthenticated request. This is the
  single most important flag for misconfiguration testing: if the bucket is properly private,
  this command fails with an `AccessDenied` error; if it's misconfigured for public listing, it
  succeeds and returns the object list.

Step 2 — If listing succeeds, test anonymous download of a specific object:

```bash
aws s3 cp s3://target-bucket-name/some-object.txt . --no-sign-request
```

- `aws s3 cp` — copy command.
- `s3://target-bucket-name/some-object.txt` — the specific object path to copy.
- `.` — destination (current local directory).
- `--no-sign-request` — same meaning as above; confirms whether `GetObject` permission is also
  open to anonymous users (listing and reading are separate permissions and can be
  misconfigured independently).

Step 3 — Test anonymous write access (only with explicit authorization/scope to do so):

```bash
echo "pentest-write-test" | aws s3 cp - s3://target-bucket-name/pentest-write-test.txt --no-sign-request
```

- `echo "pentest-write-test" | ... cp -` — the `-` tells the AWS CLI to read the object content
  from **stdin** rather than a local file, and the `echo` piped into it supplies that content.
- This should always be done with a clearly-labeled, harmless test file, and only when the
  engagement's rules of engagement explicitly permit write-testing — uploading to a client's
  production bucket without authorization can have real operational consequences.

### Automated tooling: S3Scanner

**S3Scanner** is a purpose-built tool for discovering and auditing the permission state of S3
(and S3-compatible, e.g., DigitalOcean Spaces, MinIO) buckets at scale, without needing AWS
credentials of your own.

#### Installation

```bash
pip install s3scanner --break-system-packages
```

- `pip install s3scanner` — installs the tool from PyPI.
- `--break-system-packages` — required on modern Debian/Ubuntu-based systems (including current
  Kali releases) where pip refuses to install packages into the system Python environment by
  default (PEP 668 "externally managed environment" protection); this flag overrides that
  protection for this install.

#### Basic scan against a list of candidate bucket names

```bash
s3scanner scan --buckets-file candidate-buckets.txt
```

- `s3scanner scan` — runs S3Scanner's scanning subcommand.
- `--buckets-file candidate-buckets.txt` — points S3Scanner at a text file containing one
  candidate bucket name per line. This file is typically built from your own naming-convention
  guesses (company name + common suffixes like `-backup`, `-prod`, `-dev`, `-logs`, `-assets`,
  `-static`) or generated by a permutation tool.

#### Generating candidate bucket name permutations first

```bash
s3scanner scan --bucket-file <(for suffix in backup backups prod dev staging logs assets static uploads; do echo "targetcompany-$suffix"; done)
```

- `<( ... )` — Bash **process substitution**: this runs the enclosed command and presents its
  output as if it were a file, without needing to manually create and clean up a temp file.
- The `for suffix in ...; do echo "targetcompany-$suffix"; done` loop generates one candidate
  bucket name per common suffix, piping the resulting list straight into S3Scanner's
  `--bucket-file` argument via the process substitution.

#### Interpreting S3Scanner's output

S3Scanner reports each scanned bucket name as one of: `not_exist`, `exists` (with permission
breakdown for read/write on both the bucket itself and its ACL), `auth_required` (exists but
needs credentials), or an error state. The permission breakdown it prints — typically something
like `bucket_acl: AllUsers: [READ]` — is the direct evidence of misconfiguration: any line
showing the `AllUsers` (or equivalent "anyone on the internet") group with `READ`, `WRITE`,
`READ_ACP`, or `WRITE_ACP` permissions is a finding.

#### Dumping all objects from a confirmed-open bucket

```bash
s3scanner scan --bucket-file open-buckets.txt --dump
```

- `--dump` — tells S3Scanner to download a copy of every object it finds in any bucket confirmed
  to allow anonymous read access, saving them locally for offline review (instead of just
  reporting that the bucket is open). This should only be used on in-scope targets and the
  resulting data must be handled per the client's data-handling agreement — downloaded customer
  data from a misconfigured bucket is itself sensitive and needs secure storage/deletion after
  the engagement.

### Equivalent checks for Google Cloud Storage and Azure Blob Storage

```bash
gsutil ls gs://target-bucket-name
```

- `gsutil ls` — Google Cloud's CLI list command; if run without any authenticated
  `gcloud auth login` session active, this naturally tests anonymous access, similar in spirit
  to AWS's `--no-sign-request`.

```bash
az storage blob list --container-name target-container --account-name targetaccount --auth-mode login
```

- `az storage blob list` — Azure CLI's blob-listing command.
- `--container-name target-container` — the target blob container.
- `--account-name targetaccount` — the storage account name.
- `--auth-mode login` — uses the currently logged-in Azure session for auth; testing anonymous
  access on Azure instead typically involves directly requesting the container's public URL
  (`https://targetaccount.blob.core.windows.net/target-container?restype=container&comp=list`)
  via `curl`, since Azure's public-access model is exposed as a plain unauthenticated HTTP
  endpoint when "Public access level" is enabled on the container.

## Part B: IAM Over-Permissioning and Metadata Service Exposure

### The EC2/cloud metadata endpoint pattern

Cloud compute instances expose an internal-only metadata service (commonly at
`http://169.254.169.254/`) that provides the instance with its own configuration, including, in
many setups, temporary IAM credentials. This service is only supposed to be reachable from
*inside* the instance itself — but if the application running on that instance has a
Server-Side Request Forgery (SSRF) vulnerability, or if a WAF/reverse-proxy is itself running on
an instance with overly broad IAM permissions, an attacker can use that SSRF to fetch the
instance's IAM credentials remotely. This is the exact mechanism behind the 2019 Capital One
breach referenced in file `01`.

```bash
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

- This specific request, made from *inside* a compromised request context (e.g., as the target
  parameter of an SSRF vulnerability on the actual host), lists the IAM role name attached to the
  instance. A follow-up request appending that role name to the path returns the actual temporary
  access key, secret key, and session token for that role.

This pattern is cross-referenced rather than duplicated in full detail here, since exploiting it
properly requires an SSRF bug as the entry point — covered in more depth in dedicated SSRF
material. The point of including it in this file is to frame **why IAM role scoping matters as a
misconfiguration concern**: even a perfectly patched application can be catastrophically exposed
if the IAM role attached to its hosting instance has broader permissions than the application
actually needs (violating the principle of least privilege).

## Real-world notes

- Misconfigured S3 buckets have caused dozens of large, publicly documented breaches — leaked
  voter records, leaked military contractor files, leaked customer PII from major retailers — and
  remain one of the single most reliably profitable categories in bug bounty hunting because the
  discovery technique (predictable naming + anonymous access testing) requires no advanced
  exploit development.
- Cloud Security Posture Management (CSPM) tools (AWS Config, Prowler, ScoutSuite) are the
  enterprise-side equivalent of S3Scanner — they continuously audit an organization's *own*
  cloud accounts for these exact misconfiguration patterns, which is why this category is now a
  standard line item in cloud security maturity assessments, not just an external pentest concern.
- A subtlety worth remembering for reports: a bucket can have a perfectly restrictive **bucket
  policy** but still be exposed because an individual **object's ACL** grants public access
  independently — always test object-level permissions even when the bucket-level policy looks
  correct, and vice versa.
